# Стандарты

Стандарты — правила проекта. Каждое появилось из реального опыта или ошибки. Стандарт отвечает на вопрос «зачем» и «что», скиллы — на «как».

---

## 0.x — Core (фундамент)

### 0.0 Task Master
- **Когда:** структурируешь задачу или фичу
- **Суть:** Каждая задача имеет explicit output (что создаём) и outcome (зачем это нужно пользователю)
- **Скиллы:** task-completion-persistence, document-creation-guard

### 0.1 Git Discipline
- **Когда:** любое изменение в коде
- **Суть:** После каждого важного изменения — предлагать текст коммита. Коммиты атомарные: одна фича / один фикс = один коммит. Ветки от main, PR перед merge.
- **Скиллы:** git-commit-propose, branch-review

### 0.2 TypeScript Strict
- **Когда:** любой новый файл .ts / .tsx
- **Суть:** strict: true в tsconfig. Никаких `any`. Типы для всех данных, включая ответы API. Interfaces для entities (Coin, Collection), types для утилит.

## 1.x — Process (процессы)

### 1.1 Incident Protocol
- **Когда:** произошёл баг в продакшне или сломан билд
- **Суть:** Каждый инцидент = RCA + design injection (правило, предотвращающее повтор) + фиксация в docs/incidents/
- **Обязательный скилл:** rca-incidents

### 1.2 One Mistake Rule
- **Когда:** повторяющийся паттерн ошибок
- **Суть:** Одна ошибка → одно правило → один скилл. Не допускать рецидивов.

### 1.3 Feature Lifecycle
- **Когда:** начало любой новой фичи
- **Суть:** Всегда последовательно: spec → data service → UI → SEO → code review → коммит
- **Обязательный скилл:** feature-create-sequence

### 1.4 From The End
- **Когда:** планирование фичи
- **Суть:** Начни с outcome (что пользователь делает после внедрения), двигайся назад к первому шагу

### 1.5 Gap Theory
- **Когда:** detect and close expectation-vs-fact gap
- **Суть:** Разрыв между ожиданием PM и фактической реализацией = корень проблемы. Диагностировать явно.

## 2.x — Data Layer

### 2.0 Data Service Isolation
- **Когда:** любая работа с данными
- **Суть:** UI-компоненты никогда не вызывают MCP/API напрямую. Только через `lib/services/`. Это гарантирует лёгкую замену Google Sheets на Supabase.
- **Скиллы:** service-layer-create, sheets-to-supabase-migration

### 2.1 Coin Taxonomy (иерархия монет)
- **Когда:** добавление любой монеты в базу, проектирование разделов сайта
- **Суть:** Каждая монета описывается пятью уровнями иерархии. Все поля обязательны.

**Уровень 1 — `era` (эпоха):**
| Код | Название | Период |
|---|---|---|
| `ancient` | Древняя Русь | до ~1700 |
| `empire` | Российская Империя | 1700–1917 |
| `ussr` | СССР и РСФСР | 1921–1991 |
| `rf` | Российская Федерация | 1992–нв |

**Уровень 2 — `sub_period` (подпериод):**
| Код | Для какой эпохи | Описание |
|---|---|---|
| `by-ruler` | ancient, empire | Деление по правителям/императорам |
| `1921-1958` | ussr | РСФСР и СССР, старый стандарт |
| `1961-1991` | ussr | СССР и ГКЧП, реформа 1961 |
| `1992-1993` | rf | До деноминации |
| `1997-present` | rf | После реформы 1997 |

**Уровень 3 — `type` (тип выпуска):**
| Код | Название |
|---|---|
| `regular` | Регулярные (оборотные) |
| `jubilee` | Юбилейные и памятные |
| `sets` | Наборы и коллекции |
| `regional` | Региональные выпуски |

**Уровень 4 — `material` (материал / формат) — только для `jubilee`:**
| Код | Название |
|---|---|
| `bimetal` | Биметаллические (10р) |
| `gvs` | ГВС и аналогичные (10р) |
| `silver` | Серебряные |
| `gold` | Золотые |
| `cupronickel` | Медно-никелевые |

**Уровень 5 — `category` и `series`:**
| category (код) | Название | Примеры series |
|---|---|---|
| `goroda-geroi` | Города-герои и воинской славы | `goroda-slavy`, `goroda-trudovoy-doblesti` |
| `pobeda` | Годовщины Победы в ВОВ | `pobeda-50`, `pobeda-70`, `pobeda-75` |
| `sport` | Спортивные события | `olimpiada-2014`, `chm-2018` |
| `nature` | Природа и животные | `krasnaya-kniga` |
| `culture` | Культура и искусство | `multiplikatsiya`, `arkhitektura` |
| `people` | Личности | `lichnosti` |
| `history` | Исторические события | `krymskie-sobytiya`, `pobeda-1812` |
| `industry` | Труд и промышленность | `chelovek-truda`, `oruzhie-pobedy` |
| `events` | Прочие события | `sobytiya` |

**Пример — монета «Брянск»:**
```
era: rf | sub_period: 1997-present | type: jubilee | material: bimetal | category: goroda-geroi | series: goroda-slavy
```

**URL на сайте:**
```
/rf/jubilee/bimetal/goroda-geroi/goroda-slavy/10r-2010-bryansk
```

### 2.2 Slug Standard
- **Когда:** добавление новой монеты в базу
- **Суть:** Единственный ID монеты — slug. Формат: `[номинал]-[год]-[название-латиницей]`. Используется в URL, именовании файлов, LocalStorage-ключах. Slug неизменяем после создания.
- **Пример:** `10r-2010-bryansk`

### 2.3 Image Convention
- **Когда:** добавление изображений монет
- **Суть:** Все изображения в `/public/images/coins/`. Два файла на монету: `{slug}-obverse.jpg` и `{slug}-reverse.jpg`. Оптимизация через `next/image` обязательна.
- **Порядок отображения:** Первой всегда показывается **реверс**. Аверс — после переворота.

### 2.4 Google Sheets Hard Block
- **Когда:** любая работа с Google Sheets MCP
- **Суть:** Перед написанием кода — прочитать структуру таблицы (названия колонок). Никогда не угадывать имена полей.
- **Обязательный скилл:** sheets-structure-read

### 2.5 Supabase Schema First
- **Когда:** миграция на Supabase или изменение схемы
- **Суть:** Сначала schema.sql, потом код. TypeScript-типы генерируются из схемы (`supabase gen types typescript`).

## 3.x — UI/UX

### 3.0 Design System Standard
- **Когда:** создание любого UI-компонента
- **Суть:** Shadcn/UI — базовый набор. Кастомизация только через Tailwind CSS-классы. Не устанавливать другие UI-библиотеки без явного решения. Шрифты: Inter для текста, Geist для кода.

### 3.1 Mobile First
- **Когда:** вёрстка любого компонента
- **Суть:** Начинай с мобильного (sm:), добавляй breakpoints вверх (md: lg: xl:). Сетка монет: 2 колонки на mobile, 3 на tablet, 4-5 на desktop.

### 3.2 Museum Style
- **Когда:** любое UI-решение
- **Суть:** «Современный музей» — минимализм, много воздуха, нейтральный фон, акцент на монете (изображении). Никакого overdesign. Типографика иерархична и читаема.

### 3.3 Animation Standard
- **Когда:** добавление анимаций
- **Суть:** Только Framer Motion. 3D-переворот монеты — основная анимация. Остальные: плавные переходы (ease-out, 200-300ms). Анимации не должны блокировать взаимодействие.

### 3.4 Accessibility Baseline
- **Когда:** любой UI-компонент
- **Суть:** Семантический HTML обязателен (`article`, `section`, `nav`, `main`, `button`). Alt-тексты для всех изображений. Keyboard navigation. ARIA только там, где HTML недостаточно.

## 4.x — SEO

### 4.0 SEO Standard
- **Когда:** создание любой страницы
- **Суть:** Каждая страница имеет `generateMetadata()` с title, description, Open Graph. Список страниц с требованиями:
  - `/` — главная, каталог
  - `/coins/[slug]` — страница монеты, JSON-LD Product
  - `/collections` — коллекция пользователя
- **Обязательный скилл:** seo-page-metadata

### 4.1 JSON-LD Standard
- **Когда:** страницы монет и каталога
- **Суть:** Schema.org разметка обязательна для SEO. Типы: `CollectionPage` для каталога, `Product`/`IndividualProduct` для монеты, `BreadcrumbList` на всех страницах.
- **Обязательный скилл:** jsonld-coin-schema

### 4.2 Semantic URL Structure
- **Когда:** проектирование URL
- **Суть:** URLs на основе slug. Структура: `/coins/[slug]`, `/series/[series-slug]`, `/mint/[mint-slug]`. Никаких числовых ID в URL.

## 5.x — Code Quality

### 5.0 Code Review Gate
- **Когда:** перед каждым важным коммитом
- **Суть:** code-reviewer проверяет: TypeScript ошибки, data layer отделён от UI, нет прямых вызовов MCP из компонентов, изображения через next/image.
- **Скиллы:** code-review-checklist

### 5.1 No Magic Strings
- **Когда:** любые строки-идентификаторы в коде
- **Суть:** Все константы (статусы, ключи LocalStorage, имена колонок Sheets) выносятся в `lib/constants.ts`. Никаких inline magic strings.

### 5.2 Error Handling
- **Когда:** работа с внешними данными (Sheets, Supabase)
- **Суть:** Каждый вызов данных обёрнут в try/catch. UI показывает состояния loading, error, empty. Никогда не крашить страницу из-за отсутствия данных.

## 6.x — Communication

### 6.0 PM-Dev Interaction
- **Когда:** получение задачи от PM
- **Суть:** Экспертность без лишних слов. Не объяснять базовые вещи без запроса. Сосредоточиться на ценности продукта. После выполнения — предложить текст коммита.

### 6.1 Artifact Review Standard
- **Когда:** перед показом результата PM
- **Суть:** Перед тем как показать результат — проверить глазами пользователя (client-review). Убедиться что решение соответствует дизайн-стилю «Современный музей».
- **Обязательный скилл:** client-perspective-review
